# Devlog — 2026-05-31

## One-day build, with help

Race Signals went from an empty repo to a working FEC alert desk in roughly
one day of work, all of it AI-assisted. The repo started the day with no
data ingestion and a stub homepage. By end of day it had a national multi-
state FEC ingest pipeline, a feed-first UI, four ranked entry points
(spending, fundraisers, evidence records, review queue), and a
self-healing daily/weekly cron architecture.

Two assistants did the building. Codex ran overnight in the background on
a multi-phase plan that finished most of the underlying surface area —
the Wikidata election ingest, the Schedule E evidence pages, the review
queue, the export tooling, the per-state status board, the audit scripts.
The next morning Claude Code took over for an interactive session: review,
ship the gaps, fix the bugs Codex couldn't catch from its own logs, and
talk through product direction.

## What changed today (the highlights)

**Landing experience.** The homepage opened the day as a bare feed card —
fine for a return reporter, useless for a cold one. After two iterations
it became a dark masthead with the wordmark, a one-sentence definition
of what a "signal" actually is, a three-step ingest diagram, an anatomy
callout for a sample signal card, and five CTA buttons routing into the
working surfaces. The dark band sets it apart from the white feed cards
below; on any filtered view it collapses so working reporters aren't
blocked.

**A new top entry point: `/raised`.** During user-testing as an IndyStar
congressional reporter, the question "who's raising the most in this
state" had no good answer — `/spenders` ranks outside-spending PACs, not
fundraisers. Built `/raised` as the candidate counterpart, mirroring
the spenders pattern: state and office facets, dense ranking table with
party squares, incumbent badges, funding mix percentages, and a CSV/JSON
export with the same manifest discipline as the other exports
(methodology URL, scope note, run metadata on every row).

**Ingest got real.** The morning ingest workflow was hardcoded to
Indiana — by design, for cost control, but it meant the site would
forever look like a single-state demo. Replaced `RACE_SIGNALS_STATE=IN`
with a state-agnostic sample, bumped the per-run candidate cap, and
added a separate weekly-refresh workflow that does a full per-candidate
totals re-fetch on Sundays. The daily run now skips totals fetches for
candidates whose totals were refreshed in the last week, using the
existing `totals_fetched_at` column as the gate. The same change
introduced a new requirement on the cost guard, which then needed a
follow-up commit to permit the bigger weekly cap.

**A real bug caught at the source.** A scheduled ingest the night before
had failed because the strict signal audit flagged 218 filings dated
2008 attached to a 2026 race. Indianapolis representative Andre Carson's
committee has a long FEC history. The site's existing `repair:cycles`
script could prune the bad rows, but the *root cause* was deeper. A
parallel research agent dug into FEC's OpenAPI schema and confirmed
empirically: the `report_year` parameter on `/committee/{id}/reports/`
is silently ignored. The correct parameter is `cycle` — same committee
returns 131 results with the bogus filter, 6 results with the documented
one. Same class of bug was lurking on Schedule E (using the undocumented
legacy alias `two_year_transaction_period`); fixed in the same pass with
explicit `min_date`/`max_date` bounds. The audit caught a real ingest
bug, not a data anomaly.

**Reliability.** During a manual backfill, one FEC call hung silently
for 68 minutes — the existing retry loop only retried HTTP status codes,
not network-level throws, and there was no timeout on the fetch.
Wired in `AbortSignal.timeout(60000)` plus catch-and-retry on socket
errors. Default 60 seconds, configurable via `FEC_REQUEST_TIMEOUT_MS`.

**Dead-code cleanup.** A parallel agent inventoried orphans left over
from the saved-filters feature that was deliberately cut earlier in the
day. Deleted ~250 lines across seven files (unused functions, an empty
normalization module, the `Transaction` type and its three consumers).

## The agent swarm moment

One stretch of the afternoon ran five background agents in parallel:
two on bug research (FEC `report_year` semantics, parallel Schedule E
audit), one code review of the previous hour's commits, one frontend
visual + accessibility audit across five pages, one dead-code sweep.
Two of them came back with BLOCKER-level findings the main session had
missed: the report_year bug above, and a new cost-guard regression that
would have failed every Sunday's weekly cron. Both fixed within minutes
of the reports landing. Worth doing again — the parallel review catches
things the build flow naturally won't.

## What didn't work

- First attempt at the brand mark: a single red square. User pushed back
  ("red says Republican though"). Tried again with three squares —
  red Republican, blue Democrat, emerald for fresh/money — and made
  them load-bearing across the product as a visual language.
- Pushed a workflow change that dropped the hardcoded state filter
  without realizing the cost guard required it. Caught by the code
  review agent, fixed in the next commit.
- Hung ingest process. Killed it, marked the orphan run row failed
  with an explanatory error note, then shipped the timeout fix so the
  next run can't repeat.

## What's left

Three things, none of them blockers:

1. **Cross-cycle baseline data** is not ingested. The product is 2026
   only. A reporter wanting "fundraising pace vs. 2024" would have to
   leave the site. Out of scope for the cost cap; would need the
   `transactions` table populated which we've deliberately deferred.

2. **Live election-night calls** are explicitly not in scope. Wikidata
   and Wikipedia trail real-world calls by 1-3 days. The methodology
   page is honest about this.

3. **One reporter using it.** The product hasn't been in a real
   reporter's hands yet. Everything in this devlog is what an AI
   collaborator and the project author thought was useful. The single
   most valuable thing to do next is sit one of the target users in
   front of it for 15 minutes and watch.
